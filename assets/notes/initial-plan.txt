Phase 1: Lexer

Takes the raw source code string and splits it up into tokens. Each token has
a type and a value (the raw text), plus the row and column it appeared at.

Tokens are separated by whitespaces (\n, " ", \t), that the Lexer ignores.

Gives a syntax error if it sees a character not part of the language. Each token
type has its own non-overlapping set of accepted characters, for elegant lexing.

---------------------------------------

Lexer input : The raw source code text.

Lexer output: An array of Token objects. In their original source code order.

---------------------------------------

Currently identified Token types:

    Identifier       := [a-z, _, A-Z]+   (not matching any Keywords)
    Keyword          := [PROG_END]
    OpenParenthesis  := (
    CloseParenthesis := )
    BinaryOperator   := +, -, *, /
    Semicolon        := ;
    IntegerLiteral   := [0-9]+

---------------------------------------

C++ implementation & optimization notes:

Token value is a std::string_view into a section of the imported source code
string, to avoid unnecessary heap allocations for more std::string objects,
since they are read-only (sub)strings of an already-allocated string anyway.

std::vector.reserve() is used to pre-allocate ahead of time enough memory for a
generous amount of Tokens, to avoid the hidden heap re-allocations of the
default vector memory allocation scheme in the STL, which starts the vector out
with memory for a very small number of elements and reallocates more memory
and copies all existing elements into that new heap memory block whenever adding
an element beyond the current last reserved slot. This can degrade performance.
vector.reserve() gets rid of these hidden heap allocations for most programs.

std::vector.emplace_back() is used to construct the Token objects directly into
memory slot for that vector element, which is more efficient than using
vector.push_back() which copies an already-constructed object into the vector
memory for that element.

Phase 2: Parser

The parser takes the flat array of Tokens and constructs a tree data structure
representing the source code's structure, called an Abstract Syntax Tree (AST).

It reads the eBNF grammar rules and decides whether there's a syntax error based
on these. It knows exactly what Token(s) it can expect next, based on where it's
currently at in the Token array. Semantic Rules are checked by the Parser too.

The Parser also creates the Symbol Table, which collects all used symbol names
in one place. These are names of used variables and functions. This is used to
make sure that used variables actually exist in the current scope at usage time.
It is impossible to declare a variable without initializing it, so checking
that it's present in the Symbol Table at usage time is enough for validation.

When functions, and different scopes in general, get added, the design will
necessitate having one Symbol Table per scope.

---------------------------------------

Parser input : Token array.

Parser output: 1) AST
               2) Symbol Table

---------------------------------------

Language Definition: The Language Grammar in Extended BNF form + Semantic rules.
-------------------

Language Grammar:
----------------

Program ::= Statement+ "PROG_END"

Statement ::= AssignmentStatement   (later: FunctionCallStatement, IF ELSE, etc)

AssignmentStatement ::= IDENTIFIER "=" Expression ";"

Expression ::= UINT64_LITERAL | IDENTIFIER | BinaryOperation

BinaryOperation ::= "(" Expression BinaryOperator Expression ")"

BinaryOperator ::= "+" | "-" | "*" | "/"

IDENTIFIER ::= [a-zA-Z_]+

INT_LITERAL ::= [0-9]+

Semantic Rules:
--------------

Rule 1: Each complex (sub)expression must be parenthesized. This gives the order
        which operations are carried out, no implicit operator precedence.

Rule 2: Variables must be assigned to, before being read. No declarations
        without initialization.

Rule 3: "PROG_END" signifies the end of the program. Place at source code end.

Rule 4: Each statement ends with a semicolon.

Rule 5: Unsigned 64-bit integer-only expressions & expression operands. Strings,
        negative and floating point numbers don't exist yet. Functions and
        control flow don't exist yet. Pointers don't exist yet. OS API calls
        don't exist yet.

---------------------------------------

Compiler optimizations done by walking the AST:

- Constant folding & propagation
- Replacing multiply/divide by a power of 2 with bit shifting.
- Replace multiply by 0 with 0.
- Loop-invariant code motion

---------------------------------------

C++ implementation & optimization notes:

Each AST Node is an object. There are different classes for AST Node types.
If an expression Node can contain multiple subexpressions (eg. BinaryOperation),
the class simply contains pointers to Expression Node objects by definition,
and they're set to whatever subexpressions appear in the source code during
compilation. Again, these are precisely defined by the language grammar rules.
Use regular pointers for now. Not using std::unique_ptr to avoid learning too
much C++ too quickly and overstressing my learning experience with TMI. To
store the AST itself, prefer a contiguous array of AST Node objects to improve
cache locality, instead of heap-allocating each Node with a pointer to it. Each
subexpression is strictly owned by whatever top-level expression contains it,
so if the parent expression is deleted, it must also clean up all its children.
This will be done via the destructors I suppose. std::unique_ptr can automate
this cleanup, but again, avoiding them to not be overburdening myself with
too much to learn, as I'm getting into C++ (from C) just now. If the AST Nodes
are held in a flat array of Node objects, destructors don't need to run, the
array is simply zerode out when cleanup is needed. This is elegant and ownership
isn't a thing.

The Symbol Table is stored in a std::unordered_map. We don't require any sorting
in the table, like ascending or descending order, so we don't need std::map.
The std::unordered_set and std::unordered_map are backed by a cache-friendly
contiguous hash array. You can reserve a larger memory block for it just like
with std::vector. Two different strings can lead to the same index via the hash
function, this is handled by the STL internally in ways that are a mystery to me
currently, but I assume pointer chaining or something of that nature. On the
other hand, std::set and std::map are implemented by a self-balanced binary
tree, where each tree node creation is a new heap allocation. Pointers get
changed to automatically balance the binary tree when elements are inserted.
Elements appear in ascending order by default always when iterating over the
tree. This can be swapped out for descending order by passing a template
parameter.

Phase 3: IR code generator

Map variables to stack offsets and emit the IR code for the program. SSA form
will be used for the IR. A lot of optimizations will become possible on it.

---------------------------------------

IR Generator input : Root AST Node (Program node, with its Statement nodes).

IR Generator output: IR code of the program, in Static Single Assignment form.

---------------------------------------

Compiler optimizations on the IR:

    - Common subexpression elimination
    - Auto-vectorization of loops
    - Auto-vectorization of basic blocks
    - Loop unrolling
    - Scalar evolution
    - Arithmetic operation reduction
    - Dead code elimination
    - Function inlining

Phase 4: Assembly code generator

This phase takes the IR code and emits corresponding assembly instructions for
the target architecture. Could target ARM and x64.

---------------------------------------

Assembly prologue: Emit: the .global _start directive,
                         the _start: label,
                         and standard function prologue assembly code:
                         push rbp
                         mov rbp, rsp

Traverse Statements: Walk the IR line by line and emit assembly instructions.

Evaluate Expressions via Stack:

To generate code for A + B, visit A (push its value to stack),
visit B (push its value to stack). Then emit code to:

pop rdi,
pop rax,
add rax, rdi
push rax

This is naive but guaranteed to work without complex register allocation.

Variables: When assigning a variable, evaluate the right-hand expression
           (leaving the result in rax), look up the variable's offset in
           the SymbolTable (e.g., -8), and emit:
           mov [rbp - 8], rax

Emit Epilogue / Linux Exit:

When you hit the program's only PROG_END, put the return value in rdi
(the exit code argument). Emit Linux syscall 60 (mov rax, 60, then syscall).
Without this, the Linux kernel will try to continue executing memory past your
program and throw a Segmentation Fault.

---------------------------------------

Later when I introduce a register allocator, will use: Chaitin-Briggs Algorithm.
This uses graph-coloring math to see when an available architectural register
can be used, and when spilling is needed. People say it's pretty complicated
and advanced, so I'm excited to get there.

Compiler optimizations on the assembly code:

Sliding window instruction elimination / replacing.

Phase 5: Assembly & Linking

The compiler now outputs assembly code text. This step turns that text into an
ELF binary executable file.

Function: build_executable(assembly_code: String, output_filename: String)

Input: The raw assembly string and the desired name of the executable.

Output / Purpose: Writes files to disk and triggers system build tools to
                  yield a runnable Linux binary.

Algorithm: File I/O and Subprocess Execution. Call assembler and linker.

Write the assembly_code to a file named temp.asm.

Execute the external process: nasm -f elf64 temp.asm -o temp.o
This assembles the text into an ELF object file.

Execute the external process: ld temp.o -o [output_filename]
This links the object file into a final ELF executable.

Delete the temporaries temp.asm and temp.o.
