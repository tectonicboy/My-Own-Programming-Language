OVERVIEW OF THIS DOCUMENT
=========================

All essential information about the Hirola language compiler can be found here.

This document is divided into 3 main chapters describing, respectively, the
compiler's three primary subsystems: front end, middle end and back end.
Each subsystem and its overall design, data structures and algorithms have both
their high-level design and C++ implementation described here.

A separate document describes various optimizations and contains writeups of
CPU microarchitecture-aware performance analyses conducted on the compiler.

Chapter I: Hirola Compiler Front End
--------------------------------------------------------------------------------

1. Front End Overview
=====================

The front end of the compiler carries out 2 main operations: lexing and parsing.

During lexing, the written source code is converted into objects called tokens.
The program goes from written source code statements to an array of tokens.
Examples of tokens are "{", "=", ";", keywords and variable names. Errors
are reported if the written program contains unrecognized tokens. Maintained
are the lines and columns at which each token appears, to make error reporting
by the compiler easy to understand and high-quality.

During parsing, the token array is converted into an Abstract Syntax Tree (AST).
The AST contains one node for each encountered language construct. For instance,
the binary operation (a + b) has its own AST Node that contains the sign symbol
and pointers to the AST Nodes of the two operands, which themselves might be
binary operations. Two kinds of errors are caught and reported during parsing:
syntax errors such as missing parenthesis or semicolons, and basic semantic
errors, such as reading a variable that has not yet been initialized.
Implementing the AST generating parser was complicated by the fact that the
AST is placed, node by node, inside a special memory arena, which is contiguous
to boost spatial cache locality, rather than each Node residing in its own
small dynamically allocated memory chunk. This is called data-oriented design
and improves the way modern CPUs are utilized given their microarchitecture.

A separate Semantic Analysis pass does not exist yet, but likely will later on.


2. Lexer
========

The Lexer carries out the operation of turning the written source code program
into the array of tokens. This is the primary output of the Lexer. It also has
1 secondary output as a byproduct - Auxilliary Code Block Directory, which is
simply an array of entries, where each entry gives the index of the start and
end token of each Code Block, plus the type of the Code Block.

A Code Block is a building block of the source program, is surrounded by { and }
with its start and end designated with special syntax. Examples of Code Blocks
are function definitions, the main program block, a custom data structure type
definition, (a collection of) assignments to global variables. This separation
of the input program is meant to allow multithreaded Parser operation.

Each Token object contains 4 fields:
    - a std::string_view into the input source string where the token appeared;
    - Source line and column where the token appeared;
    - The type of the token.

The Lexer is itself represented with a class. That way, multiple Lexer objects
can be spawned to process multiple input source files on multiple threads.
The primary fields of each Lexer object are the std::vector of Token objects
and one of Code_Block objects, along with the source code string. The other
fields are auxilliary and aid in the lexing operation, like current_lexeme_len.
Its only member function is what carries out the tokenizing of that Lexer's
own source code string.


3. Parser
=========

The parser constructs the AST of a given part of the input source program.

The parsing operation is implemented in the following way:

A Parsing Orchestrator is spawned, one per source file. It uses the
Auxilliary Code Block Directory populated by the Lexer to tell multiple parsers
individually which Code Blocks of the Token Array of the input source program to
parse and emit an AST and a Symbol Table of, which allows for efficient
multithreaded parsing.

The Symbol Table is held in a C++ std::unordered_map, with the key being the
actual token name string and the corresponding value being a Symbol object.
Maintaining the Symbol Table in any sorted order is not needed, thus the C++
unordered_map is better than an ordered STL map.

Symbols are represented as a class. Each Symbol object contains 3 things:
    - The kind of Symbol this is;
    - The actual symbol name string;
    - The type of the Symbol - a multi-bitfield uint64_t documented descriptor.

The AST is held in a special memory arena of bytes. Each arena slot contains one
AST Node, plus any byte offset its alignment requirements may have incurred.
A special data structure called the Auxilliary Code Block Statement Directory
contains the byte offset into the Arena of each source statement, aka of the
root of each subtree, completing the whole AST structure.

Each AST Node is represented by its own class. AST Nodes that can be of several
types are repesented by one non-instantiable abstact base class and a derived
concrete class for each of this Node's types. For example, Expression can be a
binary_operation, a literal or a symbol.

Each AST Node type has its own processor function which is called when that Node
type is encountered in the Token Array. The language grammar is made to be
unambiguous enough that the first token of any language construct (an assignment
statement, a binary_operation on the right hand side of an assignment, etc)
immediately tells the Parser what kind of AST Node this is and thus, which
processor function to call, adding much needed simplicity to the design and
implementation of AST construction.


4. Semantic analysis & error reporting
======================================

Lexing catches and reports basic syntax errors when an unknown symbol is seen,
which is not part of the defined symbols in the language grammar.

Parsing, during AST construction, catches all other syntax errors, such as
missing parentheses and semicolons, plus basic semantic errors. For now, this
only includes reporting a variable read before it has been initialized.

A fully fledged semantic analysis pass doesn't exist, but likely will later on.
