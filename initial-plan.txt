Building a compiler from scratch is hands-down one of the most rewarding and illuminating projects you can tackle in software engineering. It pulls back the curtain on how computers actually work.

To keep this achievable, we are going to design a pipeline for a minimal low-level language. Let's assume it only supports integers, basic arithmetic, variable assignment, and a return statement (which we will translate into a Linux system exit code).

Here is your architectural blueprint, broken down by pipeline stage.


================================================================================

--------------------------------------------------------------------------------

Phase 0: The Core Data Structures

Before writing functions, you need to define the data structures that will pass
through your pipeline.

Token:

A structural grouping of characters.
Contains a Type (e.g., INT_LITERAL, IDENTIFIER, PLUS, ASSIGN, RETURN)
     and a Value ("42", "x").

ASTNode (Abstract Syntax Tree Node):

Represents the hierarchical structure of your code.
Contains a NodeType (e.g., Program, BinaryOp, Assignment, Literal),
an optional Value,
and a list of child ASTNodes.

SymbolTable:

A map (dictionary) linking variable names (identifiers)
to their memory locations (e.g., stack offsets like -8(%rbp)).

--------------------------------------------------------------------------------

Phase 1: Lexical Analysis (The Lexer)

The lexer reads raw text and chops it into meaningful, categorized chunks.

Function: lex(source_code: String)

Input: The raw string of your custom language's source code.

Output / Purpose: Produces a List[Token]. It strips out whitespace and comments,
                  leaving only the syntactic vocabulary.

Algorithm: State Machine / Sliding Window.

1. Initialize a cursor at index 0 of the string.
2. Loop through the string.
3. If the current character is a letter, advance the cursor until a non-letter
   is found, extract the substring, and emit an IDENTIFIER or KEYWORD token.
4. If the current character is a digit, advance until a non-digit is found,
   and emit an INT_LITERAL token.
5. If it's a symbol (=, +, ;), emit the corresponding symbol token.
6. Advance the cursor and repeat until the end of the file.

Example tokens:

Token(Type=IDENTIFIER, Value="x")
Token(Type=INT_LITERAL, Value="42")
Token(Type=RETURN, Value="return")

The Exclusion Rule (identifiers cannot be language keywords ) is not part of
Token, it is part of the lexing decision step. The lexer's CLASSIFICATION LOGIC
has a step where it answers: Given a raw string, what token TYPE should it be?

So for example, not allowing an identifier to be the END_PROG keyword:

sequence of letters: "return"
it fits in: [a-zA-Z_][a-zA-Z0-9_]*
so the default assumption: type = IDENTIFIER
but before finalizing the token:
    if lexeme in KEYWORDS:
        type = KEYWORD_TYPE

Lexer Rule Set:
  decides Token.Type based on lexeme classification rules

--------------------------------------------------------------------------------

Phase 2: Syntax Analysis (The Parser)

The parser takes the flat list of tokens and builds a tree that represents
the grammatical logic of your code. It checks against the language definition.

Function: parse(tokens: List[Token])

Input: The List[Token] generated by the Lexer.

Output / Purpose: Produces an ASTNode representing the Root or Program node
                  of your Abstract Syntax Tree.

Algorithm: Recursive Descent Parsing.

Maintain a pointer to the "current token".

Write a function for every grammatical rule in your language
(e.g., parse_statement(), parse_expression(), parse_term()).

Start at parse_program(), which loops and calls parse_statement() until it
hits an End-Of-File token.

To parse_expression(), look for a literal or identifier. If a PLUS token
follows, recursively call parse_expression() again and wrap them both in
a BinaryOp AST node.

Crucial: If the tokens don't match your expected grammar rules,
         throw a "Syntax Error" and halt.

--------------------------------------------------------------------------------

Phase 3: Semantic Analysis

For a minimal language, this step ensures variables are declared before they
are used and maps them to stack memory.

Function: analyze_and_map(ast: ASTNode)

Input: The root ASTNode.

Output / Purpose: Produces a SymbolTable that tracks the stack offset for
                  every variable.

Algorithm: Depth-First Search (DFS) Tree Traversal.

Initialize an empty SymbolTable and current_stack_offset integer starting at 0.

Walk the AST recursively.

When you encounter a variable declaration or assignment, check if it's in
the SymbolTable.

If it is new, subtract 8 from current_stack_offset
(since x86_64 stack grows downwards and integers are 8 bytes)
and map the variable name to this new offset.

If a variable is read (used in an expression) but isn't in the table,
throw an "Undeclared Variable" error.

--------------------------------------------------------------------------------

Phase 4: Code Generation (The Emitter)

This is where the magic happens. You will translate your AST into a string
of x86_64 assembly instructions.

Function: generate_assembly(ast: ASTNode, symbols: SymbolTable)

Input: The root ASTNode and the populated SymbolTable.

Output / Purpose: Produces a String containing valid x86_64 assembly code.

Algorithm: Post-Order AST Traversal (Stack Machine Model).

Emit Prelude: Start by outputting the .global _start directive,
              the _start: label, and standard function prologue assembly code:
              push rbp
              mov rbp, rsp

Traverse Statements: Iterate through the AST children.

Evaluate Expressions via Stack:

To generate code for A + B, visit A (push its value to stack),
visit B (push its value to stack). Then emit code to:

pop rdi,
pop rax,
add rax, rdi
push rax

This is naive but guaranteed to work without complex register allocation.

Variables: When assigning a variable, evaluate the right-hand expression
           (leaving the result in rax), look up the variable's offset in
           the SymbolTable (e.g., -8), and emit:
           mov [rbp - 8], rax

Emit Epilogue / Linux Exit:

When you hit the program's only return statement, put the return value in rdi
(the exit code argument). Emit Linux syscall 60 (mov rax, 60, then syscall).
Without this, the Linux kernel will try to continue executing memory past your
program and throw a Segmentation Fault.

--------------------------------------------------------------------------------

Phase 5: Assembly & Linking

Your compiler now outputs assembly code text. You need to automate turning
that text into an ELF binary.

Function: build_executable(assembly_code: String, output_filename: String)

Input: The raw assembly String and the desired name of the executable.

Output / Purpose: Writes files to disk and triggers system build tools to
                  yield a runnable Linux binary.

Algorithm: File I/O and Subprocess Execution. Call assembler and linker.

Write the assembly_code to a file named temp.asm.

Execute an external process: nasm -f elf64 temp.asm -o temp.o
This assembles your text into an object file.

Execute an external process: ld temp.o -o [output_filename]
This links your object file into a final ELF executable.

Clean up (delete) temp.asm and temp.o.

This pipeline will successfully take a text file containing something like:

x = 5;
return x + 2;

and turn it into a Linux binary that runs and exits with a status code of 7.

--------------------------------------------------------------------------------

Language Definition: The Grammer + The set of Semantic Rules.

GRAMMAR: in Extended BNF form:
------------------------------------------------------------

Program
    ::= Statement* ReturnStatement EOF

Statement
    ::= AssignmentStatement

AssignmentStatement
    ::= IDENTIFIER "=" Expression ";"

ReturnStatement
    ::= "return" Expression ";"

Expression
    ::= INT_LITERAL
     |  IDENTIFIER
     |  "(" Expression Operator Expression ")"

Operator
    ::= "+"
     |  "-"
     |  "*"
     |  "/"

IDENTIFIER ::= [a-zA-Z_][a-zA-Z0-9_]*

INT_LITERAL ::= [0-9]+
------------------------------------------------------------

SEMANTIC RULES:
------------------------------------------------------------

Rule 1: Variables must be declared before use.

Rule 2: There must be exactly one program-ending PROG_END statement.

Rule 3: PROG_END must be the final statement in the program.

Rule 4: Integer-only expressions & expression operands. Strings don't exist yet.

--------------------------------------------------------------------------------

MINI ROADMAP:

Mini compiler spec roadmap

Phase 1 Lexer

input is raw source code string
output is ordered token stream
Lexer ignores all whitespace. Whitespace characters are <SPACE>, <TAB>, \n.

identifier symbols must be alphabetical, can have underscores too. No digits
or other special characters. Just lower / upper case ASCII english letters
and underscores.
identifiers cannot be any of the language's reserved keywords BUT this can't
be checked by the lexer. It is checked by the parser, which takes the token
stream and produces an AST while checking the token groups against the defined
language grammar.
integer literals match [0-9]+ with no signs or decimal points.
Currently the only keyword is PROG_END. Lexer doesn't care if it's present
or if it's where it's supposed to be (very end of the program). Lexer simply
parses source code strings and makes sure they are within the language.
expression operators are single character tokens = + - * /
semicolon is the single character token ;
parentheses are single character tokens ( )

Basically this reads the program and accepts all source code strings.
Each token is either the first token, the PROG_END token at EOF,
or a string in the source code surrounded by 1 or more whitespace
characters (<SPACE>, <TAB>, <\n>).
If the next token is a semicolon instead of a whitespace, that's okay,
but a space must be placed after the semicolon.
Parser categorizes the code strings and checks them against very basic rules:

Each token must be exactly one of:
        - An identifier
        - An expression operator
        - Parenthesis
        - Semicolon
        - A keyword

So now the lexer algorithm is incredibly simple to describe.

class Lexer {
private:
    std::string_view source;
    size_t cursor = 0;
    int line = 1;
    int column = 1;

public:

    // Pass source code to the constructor to initialize the state.
    //
    // source_code is a std::string that came from the system reading
    // the source code file's contents.
    // A std::string_view does not copy anything. It is just a lightweight pair
    // of a pointer and a length. It essentially says: "I point to the memory
    // owned by that string over there."
    // The Token struct can use std::string_view for its lexemes.
    // The Token stores a std::string_view that says: "This token starts at
    // pointer address 0x7fff in the file string, and is 20 characters long."
    explicit Lexer(std::string_view source_code) : source(source_code) {}

    // The member function doesn't need parameters anymore, it just runs!
    std::vector<Token> tokenize();
};

IMPORTANT: A great pillar of OOP design - having the Lexer has a class allows
           us to spin up multiple threads lexing different source files
           extremely easily, because each thread gets its own Lexer object.
           Each lexer object has well defined initialization and clean up
           of ALL the state it needs, like current cursor, line and column,
           current token type being constructed, etc. Instead of this state
           being global. An object allows us, in this case, to elegantly
           wrap all the state and functionality that a Lexer needs and have
           multiple of them running at once in a multithreaded setup lexing
           different source files, without worrying about race conditions.

A token class and a vector<Token> will exist. At start, preallocate (reserve)
enough space in the vector for 10,000 Token objects. Use tokens.emplace_back
instead of push_back, as emplace_back constructs the Token object directly
into the memory used for the vector of tokens, whereas push_back takes an
already constructed Token object (presumably constructed right before the code
that adds it to the vector) and COPIES it into the vector memory:

ALGORITHM: Lexer consumes the source code string.

1. If it's a Space, Tab, or Newline ( , \t, \n): Skip it and move on.

2. If it's a Digit (0-9): Start consuming characters until you hit a non-digit.
   Boom, you have an Integer Literal.

3. If it's a Letter/Underscore (a-z, A-Z, _): Start consuming characters until
   you hit a non-alphabetic/non-underscore one. Collect the string, and then:

   3.1. Check if it equals a reserved keyword. If yes, it's a Keyword token.
   3.2. Otherwise, it's an Identifier token.

4. If it's =, +, -, * or /

   Instantly turn it into its respective Operator token.

5. If it's ;, (, or )

   Instantly turn it into its respective Separator token.

6. Anything else: Give a "Lexer Error: Unexpected character in source code".
   Count the \n's seen so far so you can tell which line the character is on
   and at what column.

Phase 2 Parser

input is token stream
output is abstract syntax tree
program is zero or more statements followed by exactly one return
statement and EOF
return statement must be the final statement in the program
statements consist only of assignments before the return
assignment is IDENTIFIER = expression ;
left hand side must be a single identifier only
return is return expression ;
expression can be integer literal or identifier or parenthesized
expression or binary expression
binary expression is expression operator expression
operators are + - * /
parentheses override precedence
syntax errors occur on unexpected tokens missing semicolons or
invalid grammar structure
implicit precedence is multiplication and division above addition
and subtraction if implemented

Phase 3 Semantic Analysis

input is abstract syntax tree
output is validated AST with symbol table
variables must be defined before use
first assignment defines a variable
no redeclaration or shadowing allowed all variables are in a single
global scope
only 64 bit integers exist no other types exist
expressions may only use integers defined variables operators + - *
/ and parentheses
exactly one return statement must exist and it must be the final
statement
return expression defines program exit value
each variable is assigned a unique stack offset starting at -8 and
decreasing by 8 per variable
each variable occupies 8 bytes
using an undefined variable causes a semantic error
